  vs [256,256,256] 0-ply baseline:  51.2% (p=0.0006, 20k games) — stronger
  vs [512,512] 0-ply model:         51.8% (p<0.0001, 20k games) — stronger
  vs [256,256] 0-ply model:         ~52% (est from 48.0% for [256,256] vs gnubg)

Training was completely stable (56-58 samples/game throughout 500k episodes).
Speed: ~70 games/sec with 14 workers (~30x slower than 0-ply collection).

1-ply value iteration approach (inspired by jacobhilton/backgammon):
  - Compute Bellman backup at CURRENT state: target = E_dice[max_move(1-V(next))]
  - Enumerates all 21 dice outcomes, finds best move for each
  - Terminal states handled explicitly (override network with exact 0/1)
  - 0-ply move selection for game play (fast), 1-ply only for targets
  - Key insight: E[V(s')|dice,best_action] is unbiased estimator of V(s)

Key implementation details (4 bugs fixed from initial 1-ply attempt):
  1. Original used torch.max instead of torch.min (opponent picked worst moves)
  2. 1-ply move selection caused 400x slowdown + value collapse
  3. Target computed at next state instead of current state (should be Bellman backup)
  4. Terminal states not handled explicitly (network outputs ~0.4 for terminals)

Bugs found and fixed
--------------------
1. Turn encoding bug: _pick_best_play encoded resulting states with
   mover's turn instead of opponent's turn.  Fixed by adding
   switch_turn before encoding.

2. sys.path growth bug: _pick_best_play_fast called sys.path.insert
   on every move, growing sys.path to millions of entries over long
   runs.  Fixed by caching the import.

3. Move evaluation perspective bug (with perspective encoding):
   _pick_best_play encoded resulting states without switch_turn,
   causing network to evaluate "P(I win if I move again)" instead
   of "P(I win given opponent moves next)".  Fixed by adding
   switch_turn and using argmin.

4. gnubg-nn board order bug: board_to_gnubg put on-roll player in
   board[0], but gnubg convention is board[1] = on-roll player.
   Result: all gnubg-nn evaluations were inverted (P(win) returned
   as P(lose)).  This made all supervised training useless.
   Found via sanity check: gnubg-nn vs gnubg-nn gave 224 mEMG.
   User spotted that 0.449 ≈ 1 - 0.551 (gnubg-cli's value).


Summary
-------
- Perspective encoding (196 features, no turn indicator) is the best
  approach: converges faster AND reaches lower mEMG than dellalibera encoding.
- C engine: 55-61 games/sec (1.7x faster than Python's 36 games/sec).
- Lambda=0.7 does not help vs lambda=0 (TD(0)).
- lr=0.1 dominates over 0.2-0.5 in LR sweep.
- Our best TD result: 12.7 mEMG at 350k (40hu, Run 6).
- Dellalibera 150hu reference: 10.2 mEMG at 3M (plateaued).
- Dellalibera 80hu reference: 13.9 mEMG at 400k.
- gnubg supervised: plateaus at ~10 mEMG with 40hu.
- Next: scale to larger architectures ([128,128], [256,256]) with pure batch training.
- Adam optimizer (online TD): tested lr={0.001,0.0003,0.0001,0.00003}. Best 14.1 mEMG vs SGD's 13.9; 1.7x slower. Removed from td_train.py.
- ReLU 80hu TD(0): lr=0.05->0.005 over 1.15M eps -> 8.9 mEMG. Ties sigmoid 80hu head-to-head (49.5% vs 50.5%, p=0.50). Loses to gnubg-nn 0-ply 42.4%-57.6%.
- Sigmoid 80hu +2M eps (lr=0.1->0.01): beats base sigmoid 51.8%-48.2% (p=0.0003). vs gnubg-nn 0-ply: 44.2% (100k games). Both 80hu models (ReLU/sigmoid) in 42-44% range vs gnubg-nn.
- ReLU [80] anneal2 (lr=0.05->0.001, 1M eps): 7.8 mEMG. New single-layer best.
- Multi-layer results (1M eps each, from scratch):
  ReLU [80,40] lr=0.05->0.005: 7.6 mEMG. ReLU [80,80] lr=0.05->0.005: 6.9 mEMG (new project best).
  ReLU [80,60,40] lr=0.03->0.003: 7.1 mEMG. Sigmoid [80,40] lr=0.1->0.01: 10.1 mEMG.
  Multi-layer ReLU clearly beats sigmoid for deeper nets. All still improving at 1M.
- Continuation runs (+1M eps each, lr=0.02->0.001):
  ReLU [80] v3: 7.3 mEMG (3.15M total). Sigmoid [80] slightly better h2h: 50.5%-49.5% (p<0.001, 300k games).
  Both equivalent vs gnubg-nn 0-ply: sigmoid 43.8%, ReLU 43.9% (20k games each).
  ReLU [80,80] v2: 6.2 mEMG (2M total). New project best. Model: td_relu_80_80_v2_final.pt.
  Trained from scratch: 1M eps lr=0.05->0.005, then +1M eps lr=0.02->0.001.
  vs gnubg-nn 0-ply: 45.6% (20k games). Beats sigmoid [80] h2h: 51.2%-48.8% (p=0.0009, 20k games).
  ReLU [80,60,40] v2: 6.5 mEMG (2M total).
  Head-to-head (10k games): [80,80] ≈ [80,60,40] (50.7%, p=0.17). Both beat [80] significantly.
  [80,60,40] vs [80]: 52.4% (p<0.0001). [80,80] vs [80]: 51.6% (p=0.002).
  Conclusion: ReLU matches sigmoid for single-layer and dominates for multi-layer. Default to ReLU.
- v3 continuation runs (+1M eps each, lr=0.01->0.001):
  [80,80] v3: 5.8 mEMG final. Tied with v2 at 40k games (50.1%, p=0.80). Converged at 2M eps.
  [80,60,40] v3: 6.2 mEMG final (improved from 6.5). Inconclusive vs v2 (50.4%, p=0.28, 20k games).
  [80,60,40] v3 vs [80,80] v2: inconclusive (50.5%, p=0.19, 20k games). Both architectures equivalent.
- HardSigmoid [80,80] tested with online TD: 36-37 mEMG at lr=0.05 and lr=0.1. Does not work with SGD online TD.
  wildbg uses HardSigmoid with AdamW (lr=0.00113) + batch supervised training. Needs Adam + batching to avoid
  vanishing gradients in flat regions. Worth retrying when batch training is available.

Batch parameter sweep (March 2026)
-----------------------------------
Systematic tuning of batch params on [150,150] ReLU, warm-started from online TD checkpoint.

GPC sweep: tested 100-10000 on [80,80] and [150,150]. Every head-to-head inconclusive.
  GPC does not affect quality. Locked in GPC=1000.

Extended encoding experiments (March 2026)
-------------------------------------------
Tested whether additional input features improve on the base perspective196 encoding.
All experiments used ReLU [80,80] or [150,150] with warm-start from trained p196 models.

Encoders tested:
  - perspective199 (p199): p196 + 3 bounded pip count features
  - perspective203 (p203): p196 + 3 pip count + 4 bar thermometer (>=1, >=2 per player)
  - perspective370 (p370): p196 + 174 Hinton-style features (3+/4+/5+ stack indicators + bearoff thermo)

Online TD results ([80,80], 2M total episodes each, lr=0.05->0.005):
  p196 baseline: 6.8 mEMG (best 6.8 @ 1M). Baseline to beat is 6.2 from prior v2 run.
  p199:          6.3 mEMG (best @ 1.6M total). Regressed to 7.4 by 2M.
  p370:          6.5 mEMG (best @ 2M). Oscillated 7-8.5 throughout.
  p203 (warm-started from p196 [80,80]): 6.3 mEMG (best @ 800k). Oscillated 6.3-7.5.

Head-to-head (20k games each):
  p199 final vs p196 [80,80] baseline: tied (49.5%, p=0.12)
  p370 final vs p196 [80,80] baseline: tied (49.4%, p=0.11)
  p199 final vs p370 final: p199 wins (51.0%, p<0.01)

Batch TD results ([150,150], 500k episodes, warm-started from 4.5-5.0 mEMG p196 model):
  p203: 4.6-4.8 mEMG (best 4.6 @ 400k, 100-game eval).
  p203 final vs p196 baseline: tied (50.1%, p=0.70, 20k games).
  p370: 4.7-5.2 mEMG (best 4.7 @ 200k, 100-game eval).
  p370 final vs p196 baseline: tied (50.3%, p=0.43, 20k games).

Conclusion: extra features (pip count, bar thermometer, stack indicators) do NOT improve
  on the base perspective196 encoding. The network can learn these signals implicitly:
  - Pip count is a linear function of checker positions — trivially learnable
  - Bar count is already encoded (bar/2 scalar in p196)
  - Stack depth beyond 4 checkers is rare and low-impact
  The bottleneck is network capacity or training signal quality, not input features.

Phased encoding experiments (March 2026)
-----------------------------------------
Tested whether phase-gated feature copies improve on base perspective196 encoding.
The "perspective588_phased" encoder triplicates the 196 base features:
  [0-195]   always active (standard p196)
  [196-391] contact-gated copy (nonzero only in contact/crashed positions)
  [392-587] race-gated copy (nonzero only in race/bearoff positions)
Uses gnubg_nn.classify() for authoritative game-phase classification.
Rationale: gnubg trains separate networks for contact/race/crashed. Phase gating
gives a single network phase-specific behavior (W_base + W_contact or W_base + W_race)
while sharing base representations.

Also tested gnubg3gametype (p196 + 3 binary game-type features): no improvement.

Results (all warm-started from best p196 models, batch TD, Adam lr=5e-5->5e-6):

  phased588 [256,256] (1.5M eps):
    4.5 mEMG. Beats p196 [256,256] h2h 51.1%-48.9% (p=0.002, 20k games).
    Ties p196 [512,512] h2h 50.1%-49.9% (p=0.72, 20k games).
    Converged at 500k eps — extra 1M didn't improve.

  phased588 [256,256,256] (1.5M eps):
    3.9 mEMG (100-game eval). Ties p196 [256,256,256] h2h 50.2%-49.8% (p=0.28, 100k games).

  phased588 [512,512,256] (1.5M eps):
    3.6 mEMG (200-game eval). Ties p196 [512,512,256] h2h 50.0%-50.0% (p=0.91, 100k games).

  gnubg3gametype [512,512] (500k eps):
    4.5 mEMG. Ties p196 [512,512] h2h 49.9%-50.1% (p=0.76, 20k games).

Conclusion: phase gating helps smaller networks match larger ones (phased [256,256]
  matches raw [512,512]) but does not raise the performance ceiling. At each architecture
  size, the phased model ties the p196 baseline. Combined with earlier extended encoding
  experiments (p199, p203, p370 — all tied with p196), this strongly suggests the
  bottleneck is NOT input features. The base perspective196 encoding captures all
  the information the network needs. The limiting factor is likely training signal
  quality (TD(0) bootstrapping noise), network capacity, or search depth (0-ply).
